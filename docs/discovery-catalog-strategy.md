# Каталог, 2ГИС и главная страница

Сводка обсуждений: хранение данных справочника, выдача списка салонов и логика сортировки на главной.

---

## 1. 2ГИС: что хранить и что нет

### Позиция из `context.md` и `schema-analysis.md`

- Данные из ответа API 2ГИС **в долгосрочную БД не кладём** — ориентир: запросы в реальном времени, отображение из свежего ответа.
- В PostgreSQL фиксируем связку **`external_source` + `external_id`** (ID организации/фирмы в 2ГИС).
- В БД храним **свои** сущности: записи, расписание, услуги/цены, введённые салоном данные после клейма, подписки, (позже) отзывы и рейтинг **платформы**.
- Модель вроде `entity_prototype.go` — это **DTO для API**, поля «название, адрес, координаты из справочника» собираются **при ответе**: 2ГИС + слой платформы, а не обязательно колонки в БД.

### Публичные правила 2ГИС (API, в т.ч. карты)

На [law.2gis.ru/api-rules](https://law.2gis.ru/api-rules) в соглашении про API карт зафиксированы ограничения в духе:

- использование **с взаимодействием с сервисом в реальном времени**;
- **запрет** на сохранение/распространение данных, полученных через сервис, **кроме** временного кэша геокодирования **в рамках использования сервиса**.

**Коммерческий ключ** может иметь **отдельный договор** — юридически опираться нужно на свой контракт с 2ГИС, не только на публичную страницу.

### Итог по архитектуре «id + обогащение»

| Подход | Оценка |
|--------|--------|
| Запоминать **ID из 2ГИС** и привязывать к нему **свои** данные (запись, услуги, статус клейма, подписка) | **Да** |
| Держать в БД **копию** названия, адреса, координат, телефонов, часов, фото **как в ответе 2ГИС** для «не дергать API» | **Нет** при буквальном следовании описанным ограничениям (если договор не разрешает иное) |
| **Обогащение** | Данные **платформы** или **владельца** после клейма, не дублирование справочника 2ГИС |

**Выдача пользователю:** запрос к 2ГИС (гео/поиск) → список с их полями; бэкенд **дополняет** из своей БД по `external_id` (онлайн-запись, тариф, свои услуги). Кэш — только если явно разрешён договором/политикой; иначе короткий in-memory на время запроса/сессии.

## 2. Главная страница: кого показываем

### Фильтры (до сортировки)

- **Город / зона:** если клиент в Москве — показываем точки **в Москве** (границы и «Москва vs МО» — продуктовое решение, можно отдельным переключателем).
- **Геолокация:** для «рядом» и осмысленной сортировки по расстоянию нужны **координаты** или выбранный центр карты, а не только строка «Москва».
- **Рубрики:** бьюти-категории из 2ГИС / фильтры продукта.
- **Не прятать весь несетевой справочник на MVP:** иначе пустая главная; заклеймленные и с онлайн-записью можно **подсвечивать**, а не обязательно исключать остальных.

### Сортировка — этапы

**Старт (мало своих данных):**

1. **Расстояние** от пользователя (или от выбранного центра) — главный понятный критерий.
2. **Релевантность** категории/запроса.
3. Поля из **текущего ответа** 2ГИС (рейтинг/число отзывов **в 2ГИС**) — можно использовать **на лету** при отображении/сортировке, **не** складывая в БД как свой справочник.

**Когда есть данные платформы — скоринг (примерные компоненты):**

| Сигнал | Назначение |
|--------|------------|
| Расстояние | Локальность |
| Платный тариф / онлайн-запись | Монетизация и UX; **буст с потолком**, чтобы не ломать географию |
| Популярность **на платформе** | Просмотры, клики, записи за окно (7/30 дней) |
| Отзывы и рейтинг **на платформе** | Доверие; порог минимального числа отзывов от накрутки |
| Активность салона | Обновления, ответы, не «мёртвая» карточка |
| Конверсия | Показы → запись / клик |

**UI:** переключатели/вкладки вместо одной «магической» сортировки:

- **Рядом** — доминирует расстояние.
- **Популярные** — платформа + (опционально) только отображаемые сигналы из ответа 2ГИС без долгого хранения.
- **По рейтингу** — имеет смысл, когда накоплены **свои** отзывы; до этого вкладка слабая → можно не показывать или «скоро».

### Дополнительные идеи

- Небольшой буст **новым** салонам на платформе первые N дней.
- Позже: **«Свободно сегодня»** по слотам расписания.
- Персонализация по истории записей и любимым категориям (с согласия).
- Приоритет карточкам с **заполненным профилем** после клейма (фото, услуги).

### Ограничение по 2ГИС

Долгосрочный «рейтинг в нашей БД», построенный на **скопированных** данных 2ГИС, — нежелателен. **Своя** популярность и **свои** отзывы — только события и таблицы платформы.
