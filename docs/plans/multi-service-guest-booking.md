# План: мульти-услуга в гостевой записи + модель цен и сводка над формой контактов

> Монорепо `beauty-marketplace`. После реализации: `go test ./...`, `npm run lint`, `npm run build`.  
> Итоговая сводка реализации и список улучшений: [`docs/multi-service-booking-rollout-summary.md`](../multi-service-booking-rollout-summary.md).

---

## 1. Цели продукта

1. Гость на **странице салона** может выбрать **одну или несколько** услуг, затем мастера (только те, кто оказывает **все** выбранные услуги **в этом салоне**), затем время **одним непрерывным блоком**; суммарная длительность не пересекается с другими записями мастера.
2. **Цена** для визита считается по согласованным правилам (см. §3) и показывается **не на первом экране выбора услуг**; блок с ценой, временем и мастером — **над формой** имени/телефона/комментария на последнем шаге перед отправкой (см. §4).
3. Разделить смыслы цен: **каталог салона**, **оверрайд владельца салона на мастера**, **(опционально) цена/канал через публичный профиль мастера** — явно зафиксировать в данных и API, чтобы не смешивать «профиль» и «салон».

## 2. Анализ текущего кода (факт)

### 2.1 Откуда сейчас берётся цена на странице салона

Публичный список мастеров салона: `GET /api/v1/salons/:salonId/masters` → [`ListSalonMastersPublic`](backend/internal/service/master_public.go).

Для каждой связки мастер–услуга в салоне используются строки `salon_master_services` + цена из каталога услуг салона `services.price_cents`:

- **База:** `services.price_cents` для услуги в этом салоне (одна цена «у салона» на услугу — соответствует твоему сценарию **3**).
- **Оверрайд владельца салона на конкретного мастера:** `salon_master_services.price_override_cents` → в DTO поля `priceOverrideCents` и **`effectivePriceCents`** = `effectivePriceCents(salonPrice, override)` — сценарий **2**.
- Длительность аналогично: `services.duration_minutes` + `salon_master_services.duration_override_minutes`.

Итого для **гостевой записи с SalonPage** расчёт «сколько стоит у этого мастера эта услуга в этом салоне» **уже определён**: брать **`effectivePriceCents`** / эффективную длительность из ответа `GET .../masters` (или дублировать ту же формулу на бэке при `POST /bookings`, см. §5).

### 2.2 Публичный профиль мастера (сценарий **1**)

`GET /api/v1/masters/:masterProfileId` → [`GetMasterProfilePublic`](backend/internal/infrastructure/persistence/master_public_repository.go): услуги по членствам всё равно читаются через **`salon_master_services` + `services` того же салона** (те же `salon_price_cents` и `price_override_cents`). В карточке салона на странице мастера в API сейчас в основном **названия** услуг (`MasterSalonCardDTO.Services []string`), а не отдельная «личная прайс-листа мастера, независимая от салона».

**Разрыв:** отдельного хранилища «цена услуги только для записи через профиль мастера, не как в салоне» в схеме **пока нет**. Чтобы выполнить сценарий **1** буквально («цена в профиле ≠ цена у салона для того же мастера»), нужно одно из:

- **A)** новые сущности/поля (например `master_profile_service_offers`: `master_profile_id`, `service_id` или абстрактная услуга, `price_cents`, `visible_on_profile`) и отдельный расчёт при `booking_source=master_profile`; или  
- **B)** продуктово решить, что «запись через профиль» для услуг салона **всё равно** использует цену салона/оверрайда (сценарии 2–3), а «личная цена» касается только услуг вне салона (другой тип сущности, позже).

План ниже закладывает **канал бронирования** (`salon_page` | `master_profile`) и **серверный прайсинг**, чтобы вариант **A** можно было включить без ломки контракта.

## 3. Три сценария цен (как их учесть)

| # | Сценарий | Смысл | Реализация в данных (целевое) |
|---|----------|--------|--------------------------------|
| **1** | Запись **через публичный профиль мастера** | Цена может задаваться мастером для «своего» канала | Сейчас: частично не отделено от салона — нужна схема **A** из §2.2 или явный MVP: цена как у салона для выбранного салона в карточке. |
| **2** | Страница салона, цена **на мастера** задана владельцем | Отличается от других мастеров и от «голого» прайса услуги | Уже есть: `salon_master_services.price_override_cents` + база `services.price_cents` → `effectivePriceCents`. |
| **3** | Страница салона, **одна** цена услуги на всех | Общий прайс салона | Уже есть: только `services.price_cents`, оверрайд NULL. |

**Правило приоритета для визита (после выбора салона и мастера в контексте салона):**

1. Если канал = `master_profile` и для пары (услуга, мастер) есть **профильная** цена (после внедрения схемы A) → использовать её.  
2. Иначе если есть `salon_master_services.price_override_cents` → использовать оверрайд поверх `services.price_cents`.  
3. Иначе → `services.price_cents`.

Для **мульти-услуги**: итог = **сумма** эффективных цен по каждой выбранной услуге для **выбранного** `salon_master_id` (и канала). Порядок услуг на цену суммирования не влияет; на длительность — сумма минут (см. трек слотов).

## 4. UX модального окна (обновление относительно прежнего плана)

**Не показывать цену** на первом шаге (мультивыбор услуг): только названия, длительность (опционально без цены — по желанию продукта), чекбоксы и кнопка «Выбрать дату и мастера» (disabled, пока не выбрана хотя бы одна услуга).

Рекомендуемая последовательность шагов:

1. **Услуги** (мультивыбор, без суммарной цены или только «~N услуг» без денег — по продукту).  
2. **Мастер** (пересечение по `salon_master_services` / DTO `services` с `GET /salons/:id/masters` — **не** опираться на услуги только из маркетингового текста профиля без привязки к салону).  
3. **Дата и слот** (`PublicSlotPicker` / расширенный API под суммарную длительность).  
4. **Контакты и запись** — **один экран**: сверху блок **итога** (не отдельный шаг после формы): выбранный **мастер**, **дата и время** слота, список услуг с **ценой по строке** (по §3) и **итоговая сумма**; **ниже** — поля **имя, телефон, комментарий**; внизу кнопка **«Записаться»** → `POST`. Гость сначала видит сводку с ценой и временем, затем заполняет контакты и подтверждает.

Так цена не попадает на первый шаг, а отображается **над формой контактов** на последнем шаге перед фиксацией.

## 5. Бэкенд: расчёт суммы и защита от рассинхрона

- Ввести (в рамках задачи) **серверный** расчёт итога: например `POST /api/v1/salons/:id/booking-quote` с `{ serviceIds[], salonMasterId|masterProfileId, channel }` → `{ lines[], totalCents, totalMinutes }` **или** включить те же поля в ответ `GET .../slots` как `meta` (тяжелее кэшировать). Минимум: при **`POST /bookings`** пересчитать цены и сравнить с опциональным `clientDeclaredTotalCents` (если передаётся) или просто вернуть в ответе созданной записи зафиксированные суммы (**snapshot** в БД — см. §6).
- **Снимок цен** в момент бронирования обязателен: цены в каталоге могут измениться после показа итога; в строках визита хранить `price_cents_snapshot` (и длительность) на услугу.

## 6. Хранение визита с несколькими услугами (как в предыдущей версии плана)

**Вариант A (рекомендуется):** одна запись `appointments` + таблица **`appointment_line_items`** (`appointment_id`, `service_id`, `sort_order`, `duration_minutes_snapshot`, `price_cents_snapshot`). Одна полоса времени на календаре, один `appointmentId`.

**Вариант B:** несколько `appointments` с `booking_group_id`, времена подряд. Проще миграция, сложнее UX календаря.

Рекомендация: **A**, если важна одна сущность «визит» и один чек для гостя.

---

## 7. Слоты (мульти-услуга)

- Расширить `GetAvailableSlots` / `SlotParams`: несколько `service_id` → суммарная длительность по выбранному мастеру (каждая услуга: база + `duration_override` из `salon_master_services`).
- Конфликт: интервал `[startsAt, endsAt)` той же длины против существующих `appointments` этого `salon_master_id`.

## 10. Порядок работ (крупными этапами)

1. **Продукт:** зафиксировать MVP для сценария **1** (вариант A с отдельными ценами профиля **или** явный «как у салона до фазы 2»).  
2. **Схема БД:** `appointment_line_items` (+ при необходимости таблица/поля под цены профиля) + миграция.  
3. **Прайсинг-сервис** на бэке: функция `ResolveLinePrice(ctx, channel, salonID, salonMasterID, serviceID)` с правилами §3.  
4. **Слоты** под несколько услуг + тесты.  
5. **POST /bookings** + snapshot + транзакция.  
6. **Фронт:** wizard; на последнем шаге сводка (мастер, время, цены) **над** полями контактов; без цены на шаге 1.  
7. **(Опционально)** Запись с `MasterPage`: тот же диалог/другой entry с `channel=master_profile`.

## 11. Риски

- **Дашборд:** список записей и фильтр по услуге для мульти-визита закрыты (подпись из `appointment_line_items`, фильтр `service_id` с `EXISTS`). Остаётся, например, привязка **отзыва** к визиту с несколькими услугами — отдельное решение.  
- Без серверного пересчёта и snapshot возможна смена цен между шагами и спор с гостем (для гостевого флоу snapshot в `appointment_line_items` уже пишется при бронировании).

## 12. Дашборд и кабинет мастера (отображение)

После появления `appointment_line_items`:

- В ответах списков записей поле **`serviceName`** может содержать несколько названий через запятую (порядок = `sort_order` в line items). Если строк в `appointment_line_items` нет (старые записи или создание из дашборда без line items), подпись как раньше — из **`services.name`** по `appointments.service_id`.
- Фильтр **`service_id`** в query списка записей салона (`GET .../dashboard/appointments`) отбирает записи, где `appointments.service_id = service_id` **или** существует строка в **`appointment_line_items`** с тем же `service_id` (мульти-визит). Реализация: `dashboard_repository.ListAppointments`.
