Детальный разбор
5 стратегических направлений
Каждое направление — детальный анализ: что делаем, зачем, как, какие риски, и какие метрики покажут успех.
01

Telegram Mini App

Живи там, где уже живёт аудитория

✦ Преимущества

  • 80%+ целевой аудитории уже в Telegram — нулевой барьер входа
  • Auth решается нативно через Telegram Login — не нужна OTP-доставка
  • Виральность через шеринг в чатах: «записалась через вот это»
  • Push-уведомления бесплатно и мгновенно — нативный канал
  • Mini Apps поддерживают React — можно переиспользовать 60-70% фронта
  • Telegram Payments (Stars) — встроенный платёжный механизм

▲ Риски и сложности

  • Ограничения UI: нет полноценного URL-роутинга, ограниченная навигация
  • Нужно адаптировать весь UX под мобильный формат Mini App
  • Dashboard салона слишком сложен для Mini App — нужен отдельный веб
  • Зависимость от одной платформы: Telegram может изменить правила
  • SEO отсутствует — весь трафик надо нагонять вручную
  • 3–5 недель на адаптацию фронтенда + тестирование

⚙ Что нужно сделать

  • Telegram Bot + Mini App setup (BotFather, webhook)
  • Адаптировать SearchPage → мобильный Mini App интерфейс
  • Telegram Login Widget → заменяет OTP полностью
  • Notification pipeline: бот → пользователю при букинге
  • Адаптировать GuestBookingDialog → компактный wizard
  • Deep links для шеринга: t.me/beautica_bot/app?startapp=salon_xxx

◎ Метрики успеха

  • MAU Mini App — рост месяц-к-месяцу
  • Booking conversion — % от открытия до записи
  • Viral coefficient — сколько новых юзеров от одного шера
  • Retention D7/D30 — возвращаемость
  • Bot → Mini App funnel — % от старта бота до действия
Ключевой инсайт: Telegram Mini App — это не просто канал дистрибуции, это смена парадигмы. Ты перестаёшь конкурировать за внимание в браузере и начинаешь жить внутри мессенджера. В России это особенно мощно: VK Mini Apps уже показали, что «приложение внутри мессенджера» — привычный паттерн для аудитории. Но ceiling выше, потому что Telegram растёт, а VK стагнирует.
02

Booking Link для мастеров

Calendly для бьюти — wedge-стратегия

✦ Преимущества

  • Решает конкретную боль: мастера записывают клиентов в личку / блокнот
  • Supply и demand приходят одновременно — мастер приводит своих клиентов
  • Переиспользует 90%+ существующего кода: /master/:id, слоты, букинг
  • Самая быстрая итерация к первым реальным юзерам (1–2 недели)
  • Органический рост маркетплейса: мастера → клиенты → другие мастера
  • Легко продавать: «У тебя нет записи? Вот бесплатная ссылка»

▲ Риски и сложности

  • Маркетплейс может не вырасти из набора изолированных ссылок
  • Конкуренция: tap.link, mssg.me и аналоги уже в нише
  • Монетизация неочевидна на старте — инструмент должен быть бесплатным
  • Нужен onboarding-сценарий для мастера без знания платформы
  • Без SEO и маркетплейса — ограниченный ceiling для клиентской стороны

⚙ Что нужно сделать

  • Упростить /master/:id → лендинг с услугами, ценами, слотами
  • Vanity URL: beautica.ru/nastya (slug для мастера)
  • Self-service регистрация: мастер → профиль → расписание → ссылка
  • Мини-дашборд мастера: входящие записи, расписание, уведомления
  • Шеринг-карточка (OG:image) для Instagram/Telegram bio
  • Telegram-уведомление мастеру при новой записи

◎ Метрики успеха

  • Masters registered — сколько мастеров создали ссылку
  • Link → Booking rate — конверсия посещения ссылки в запись
  • Bookings/master/week — активность через платформу
  • Cross-discovery — % клиентов, записавшихся к другому мастеру
  • NPS мастера — порекомендует ли коллеге
Ключевой инсайт: Это классическая «wedge» стратегия: входишь с маленьким инструментом, который решает конкретную боль, а потом расширяешься в платформу. Так делал Calendly (ссылка на запись → полная система расписаний), так делал Shopify (магазин → экосистема). Мастер приходит за ссылкой, а потом обнаруживает аналитику, отзывы, виджет для Instagram — и платит за это.
03

Агрегатор YCLIENTS / DIKIDI

Стань Aviasales для бьюти

✦ Преимущества

  • Мгновенное решение cold start: 80k+ салонов DIKIDI сразу в каталоге
  • Реальные слоты и цены — не пустые профили из 2GIS
  • Можно показать работающий продукт без единого онборженного салона
  • Конкурентный моат: агрегация сложна, но даёт network effect
  • Монетизация через лидогенерацию: плата за приведённого клиента

▲ Риски и сложности

  • API YCLIENTS и DIKIDI — платные, с ограничениями, могут закрыть доступ
  • Юридический риск: парсинг без разрешения = проблемы
  • Зависимость от чужих данных: качество, актуальность, формат
  • Самые большие затраты на разработку из всех направлений
  • YCLIENTS может запустить свой маркетплейс (YPLACES) и перекрыть API
  • Сложная синхронизация: расписания меняются в реалтайме

⚙ Что нужно сделать

  • Исследовать API YCLIENTS: стоимость, лимиты, разрешения
  • Исследовать API DIKIDI: аналогично
  • Адаптер по аналогии с PlacesProvider для каждой CRM
  • Unified search: 2GIS (каталог) + YCLIENTS/DIKIDI (слоты/цены)
  • Обратный букинг: запись через API провайдера
  • Reconciliation: обработка конфликтов и отмен

◎ Метрики успеха

  • Coverage — % салонов Москвы с реальными слотами
  • Booking success rate — % записей без ошибок синхронизации
  • Cost per booking — стоимость API-запросов на одну запись
  • Search → Booking funnel — конверсия поиска
  • Data freshness — задержка обновления расписаний
Ключевой инсайт: Агрегаторная модель — самая мощная для решения cold start, но и самая рискованная для solo-dev. Зависимость от чужих API — стратегическая уязвимость. Имеет смысл как второй этап: после того как Booking Link привлёк первых мастеров, агрегация YCLIENTS/DIKIDI расширяет каталог для клиентов.
04

Локальный blitz — один район

Не масштабируется. Зато валидирует.

✦ Преимущества

  • Максимально быстрая валидация product-market fit
  • Прямой контакт с салонами — лучший источник фидбека
  • Развивает навык продаж — главный identified risk
  • Плотное покрытие одного района = убедительная демо для инвесторов
  • Минимум дополнительной разработки — работа «в поле»

▲ Риски и сложности

  • Не масштабируется: каждый район = ручная работа
  • Требует много личного времени при наличии основной работы
  • Без OTP-доставки и деплоя невозможен (зависимость от отложенных задач)
  • Один район может не дать статистически значимый сигнал
  • Психологически тяжело: холодные продажи без опыта

⚙ Что нужно сделать

  • Подготовить промо-материалы: QR-наклейки, буклеты
  • Скрипт первого контакта (elevator pitch для салона)
  • Лендинг для салонов: «Подключитесь бесплатно»
  • Выбрать район с высокой плотностью бьюти-бизнесов
  • Гео-таргет рекламы ВК/Telegram на этот район
  • Feedback-петля: Telegram-чат с подключёнными салонами

◎ Метрики успеха

  • Salon sign-up rate — % салонов, согласившихся в районе
  • First booking time — время от подключения до первой записи
  • Salon retention W4 — используют ли через месяц
  • Client NPS — опрос после первой записи
  • Objection log — почему салоны отказывают
Ключевой инсайт: Локальный blitz — это не продуктовая стратегия, это GTM-эксперимент. Его ценность не в пользователях, а в обучении: ты узнаешь, как говорить с салонами, какие возражения они дают, и что на самом деле им нужно. Для solo-dev без sales-опыта это может быть самый ценный инвестированный месяц. Но он зависит от деплоя, поэтому это не «прямо сейчас».
05

White-label SaaS для мини-сетей

Продавай дашборд, не маркетплейс

✦ Преимущества

  • Понятный B2B revenue: подписка за инструмент управления
  • Dashboard уже мощный: календарь, мастера, DnD, услуги
  • Мини-сети (2–5 филиалов) — underserved сегмент: YCLIENTS дорого, 1C сложно
  • Каждая сеть = 10–30 мастеров сразу, меньше онбординга на мастера
  • Стабильный recurring revenue vs комиссионная модель

▲ Риски и сложности

  • Это другой бизнес: B2B SaaS ≠ маркетплейс, другие навыки продаж
  • YCLIENTS — 800-lb gorilla в этой нише с 10+ лет развития
  • Нужна мультитенантность: несколько салонов под одним аккаунтом
  • Feature requests от B2B клиентов могут утянуть в кастомизацию
  • Отвлекает от маркетплейсной модели — основной vision

⚙ Что нужно сделать

  • Multi-salon view в dashboard: переключение между филиалами
  • Агрегированная аналитика по сети (cross-salon reporting)
  • Централизованное управление мастерами и услугами
  • White-label: своё лого, домен, цвета
  • Биллинг: Stripe / ЮКасса, подписочная модель
  • Onboarding wizard для сети

◎ Метрики успеха

  • MRR — ежемесячный recurring revenue
  • Churn rate — уход клиентов/месяц
  • Seats per account — мастеров на одну сеть
  • Feature adoption — какие фичи используют
  • Support load — тикетов на клиента в месяц
Ключевой инсайт: White-label SaaS — привлекательный pivot из-за уже готового дашборда, но это стратегическая ловушка на данном этапе. Ты будешь конкурировать с YCLIENTS и DIKIDI на их территории (B2B инструменты), где у них 10+ лет headstart. Маркетплейс — территория, где их нет. Имеет смысл как monetization layer позже, когда база мастеров позволяет предложить управление «мини-сетью» из маркетплейса.

Risk Assessment
Ключевые риски
Сквозные риски, актуальные для любого из направлений, и специфичные для каждого.
Критический

Zero-user risk

Продукт технически зрел, но не видел ни одного пользователя. Все гипотезы — предположения. Каждый день без деплоя = день без обучения. Mitigation: Booking Link как fastest path to user.

Критический

Solo-dev bottleneck

Один разработчик с основной работой. Любое направление с длинным dev-циклом (Агрегатор, SaaS) рискует растянуться в 2–3 раза. Mitigation: выбирать направление с минимальным dev overhead.

Средний

GTM inexperience

Нет опыта продаж и маркетинга. Направления, требующие cold outreach (Blitz, SaaS), несут повышенный риск. Mitigation: Booking Link — inbound-подход, мастер приходит сам.

Средний

Platform dependency

Telegram Mini App = зависимость от Telegram; Агрегатор = зависимость от API YCLIENTS/DIKIDI. Mitigation: web-версия как fallback, собственная БД мастеров.

Низкий

Competitive response

YCLIENTS может ускорить YPLACES; DIKIDI может запустить маркетплейс. Окно возможностей 12–18 мес. Mitigation: скорость, фокус на мастерах (у конкурентов B2B-фокус).

Низкий

Tech debt accumulation

Быстрый pivot между направлениями может создать технический долг. Mitigation: текущая архитектура Clean Architecture + PlacesProvider interface изоляция.

◆ Рекомендация

Начинай с Booking Link. Это кратчайший путь от кода к первым реальным записям.

У тебя уже есть 90% нужного кода. Booking Link максимально переиспользует /master/:id, PublicSlotPicker, и гостевую запись. Мастер приводит своих клиентов — ты получаешь supply и demand одновременно, без cold outreach к салонам. А маркетплейс вырастает органически из базы мастеров. Telegram Mini App — следующий шаг, который даёт distribution. Вместе они образуют flywheel: мастер шерит ссылку → клиент записывается → клиент находит других мастеров в Mini App → новый мастер приходит за своей ссылкой.
