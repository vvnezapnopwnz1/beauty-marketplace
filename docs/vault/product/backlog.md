# Backlog

Список фич и улучшений, сознательно отложенных на будущие итерации (чтобы не терялись). Добавляй сюда пункты из брейнштормов и дизайн-сессий. Короткая формулировка + ссылка на спеку / тикет, если есть.

## Профиль пользователя

- **Смена телефона через OTP** — редкая операция, нетривиальный флоу (OTP на новый номер, проверка занятости, опционально OTP на старый + logout everywhere). Сейчас в UI просто отображаем телефон без возможности менять. Отложено из спеки `2026-04-24-user-profile-design.md` (вопрос 4 → вариант A). Когда делать: когда появятся реальные юзеры и запросы из поддержки.
- **Загрузка аватара пользователя** — сейчас `avatarUrl` хранится как простая строка (юзер вставляет ссылку). Нужен эндпоинт `POST /api/me/avatar` (multipart), серверное хранилище и ресайз превью. Предпочтительный путь — **S3-совместимое хранилище** (Yandex Object Storage в prod, MinIO в dev через `docker-compose.yml`), чтобы не привязываться к локальному диску контейнера; альтернатива на время — локальный диск с volume-маунтом. Эта же инфраструктура понадобится для фото салонов, мастеров, услуг и отзывов — делать один раз. Отложено из спеки `2026-04-24-user-profile-design.md` (вопрос 5 → вариант D). Когда делать: перед первой публичной загрузкой контента (фото салона в кабинете).
- **Деприкация `GET /api/auth/me`** — фронтенд переехал на `/api/v1/me`, но старый эндпоинт оставлен для обратной совместимости. Нужно спланировать удаление: инвентаризация оставшихся вызовов, релиз-ноты, срок отключения и удаление кода в `auth`-контроллере/клиентах. Когда делать: после стабилизации `/me` и одного релизного цикла без обращений к старому маршруту.

## Роли и права

- **Расширить `salon_members.role`** — добавить `receptionist` (только запись клиентов, без редактирования услуг/мастеров) и `accountant` (только отчёты/статистика). Требует: миграция enum `salon_member_role`, обновление гейтов в `DashboardService`, UI в «Сотрудники» для назначения роли. Отложено из `2026-04-24-user-profile-design.md` (вопрос 3 → вариант A). Когда делать: когда салоны начнут просить «разные уровни доступа для ресепшен и бухгалтера».
- **Super-admin платформы** — роль `admin` в `global_role` уже есть, но нет интерфейса и API под неё (модерация салонов, блокировка пользователей, глобальная статистика). Когда делать: при онбординге первых платных салонов, чтобы иметь инструменты поддержки.
