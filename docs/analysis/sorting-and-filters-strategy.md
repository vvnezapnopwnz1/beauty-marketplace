# Стратегия сортировки и фильтрации в поиске

> Дата: 2026-04-09
> Контекст: SearchPage уже работает через `GET /api/v1/search` → 2GIS. Сейчас сортировка только по расстоянию. Нужно добавить сортировку по популярности/рейтингу и пять фильтров.

---

## 1. Текущее состояние

Как работает поиск сейчас:

1. Фронтенд отправляет `lat/lon/category/page` на `GET /api/v1/search`
2. `SearchService` транслирует категорию в текстовый запрос, вызывает 2GIS через `PlacesProvider.SearchNearby`
3. 2GIS возвращает результаты (по умолчанию `sort=relevance`)
4. `SearchService` фильтрует шум, матчит с внутренними салонами через `salon_external_ids`, подмешивает услуги, считает расстояние по Haversine
5. Фронтенд отображает карточки

**Проблема:** пользователь не может выбрать, что ему важнее — близость, рейтинг или популярность. Фильтры на фронте частично визуальные заглушки.

## 2. Сортировка: три режима

### Что умеет 2GIS API

| Наш режим | Параметр 2GIS | Что делает |
|-----------|---------------|------------|
| **Рядом** | `sort=distance` + `point=lon,lat` | Строго по расстоянию от пользователя |
| **Популярные** | `sort=relevance` + `location=lon,lat` | Микс: совпадение имени + популярность + рейтинг + расстояние + реклама 2GIS |
| **По рейтингу** | `sort=rating` | Строго по убыванию рейтинга 2GIS |

### Рекомендуемая логика

**«Популярные» — дефолт.** Новый пользователь открывает приложение и видит не просто ближайшие подвалы, а релевантную выдачу, где 2GIS уже учёл популярность и качество. Это лучший первый опыт.

**«Рядом» — переключение.** Пользователь в спешке, ему нужно буквально ближайшее место. Тут `sort=distance` с жёстким `radius`.

**«По рейтингу» — переключение.** Пользователь готов проехать дальше ради качества. `sort=rating` без жёсткого радиуса (или с большим, 5–10 км).

### Как это влияет на запрос к 2GIS

```
# Популярные (default)
sort=relevance & location=lon,lat & radius=5000

# Рядом
sort=distance & point=lon,lat & radius=3000

# По рейтингу
sort=rating & point=lon,lat & radius=10000
```

Ключевая разница: для `sort=distance` обязателен `point`, для `sort=relevance` лучше использовать `location` (мягкое смещение, не ограничивает область). Для `sort=rating` — `point` с большим радиусом, чтобы захватить топовые салоны в районе.

### Реализация

**Backend:** добавить query-параметр `sort` в `GET /api/v1/search`:

```
GET /api/v1/search?lat=55.75&lon=37.62&category=nails&sort=popular
```

Допустимые значения: `popular` (default), `nearby`, `rating`.

`SearchService` мапит:
- `popular` → 2GIS `sort=relevance`, `location=lon,lat`
- `nearby` → 2GIS `sort=distance`, `point=lon,lat`
- `rating` → 2GIS `sort=rating`, `point=lon,lat`

**Frontend:** `searchSlice` уже имеет `sortBy`. Подключить его к реальному запросу. В UI — три чипа или сегмент-контрол.

## 3. Фильтры: пять штук, три источника данных

### Карта фильтров по источнику

| Фильтр | Источник данных | Где фильтруется | Параметр |
|--------|----------------|-----------------|----------|
| **Открыто сейчас** | 2GIS API | На стороне 2GIS | `work_time=now` |
| **Высокий рейтинг** | 2GIS API | На стороне 2GIS | `has_rating=true` + post-filter `rating >= 4.5` |
| **Онлайн-запись** | Наша БД | Post-filter после enrichment join | `salon.online_booking_enabled` |
| **Свободно сегодня** | Наша БД (расписание + записи) | Post-filter | Проверка свободных слотов |
| **До 2000 ₽** | Наша БД (таблица `services`) | Post-filter | `min(services.price_cents) <= 200000` |

### Критический момент: два класса фильтров

**Класс A — «фильтры 2GIS»** (Открыто сейчас, Высокий рейтинг):
Добавляются как параметры к запросу 2GIS. Результатов будет меньше, но пагинация работает нормально — 2GIS сам отфильтровал лишнее.

**Класс B — «фильтры платформы»** (Онлайн-запись, Свободно сегодня, До 2000 ₽):
Применяются **после** получения результатов из 2GIS. Это принципиально другая механика, потому что:

1. Большинство результатов 2GIS **не имеют** привязки к нашей БД (нет записи в `salon_external_ids`)
2. Если фильтр платформы активен — результат будет показан **только** если салон зарегистрирован на платформе
3. Пагинация ломается: запросил 20 у 2GIS, после post-filter осталось 3

## 4. Архитектура фильтрации

### Принцип: фильтры класса B переключают режим поиска

Когда активен **любой** фильтр класса B, поиск работает иначе:

```
Без фильтров класса B:
  2GIS (20 результатов) → enrichment → показ всех 20

С фильтром класса B:
  2GIS (50 результатов) → enrichment → post-filter → показ N из 50
  Если N < target: запросить ещё страницу у 2GIS, повторить
```

**Почему 50, а не 20:** при post-filter нужен запас. Если из 20 результатов только 2 зарегистрированы на платформе — пользователь увидит 2 карточки. Нужно over-fetch.

### Алгоритм для фильтров класса B

```
target_count = page_size (20)
max_pages = 3  // защита от бесконечного цикла
collected = []

for page in 1..max_pages:
    raw = fetch_from_2gis(page, page_size=50, filters_class_a)
    enriched = enrich_with_platform_data(raw)
    filtered = apply_platform_filters(enriched, filters_class_b)
    collected.append(filtered)
    
    if len(collected) >= target_count:
        break
    if raw.total <= page * 50:
        break  // 2GIS выдал всё что есть

return collected[:target_count]
```

**Лимит max_pages=3** важен: не позволяем одному запросу пользователя генерировать 10+ запросов к 2GIS. Лучше показать меньше результатов с честным сообщением «Показано 7 из возможных», чем тормозить 5 секунд.

### UX при малом количестве результатов

Если фильтры класса B включены и результатов мало (< 5), показать:

> «Мало результатов с онлайн-записью рядом. Попробуйте увеличить радиус или убрать фильтр.»

Это честнее, чем пустая выдача, и мотивирует салоны подключаться к платформе.

## 5. Детальная логика каждого фильтра

### 5.1 Открыто сейчас

**Источник:** 2GIS API, параметр `work_time=now`.

**Логика:** 2GIS проверяет расписание организации и возвращает только те, что открыты в момент запроса. Учитывает таймзону региона.

**Реализация:** просто добавить `work_time=now` к запросу. Никакой post-обработки.

**Нюанс:** это фильтр по часам работы заведения, а не по наличию свободных слотов. Парикмахерская может быть открыта, но все мастера заняты. Для MVP это допустимое упрощение — «открыто» ≠ «свободно», но пользователь в 80% случаев понимает это интуитивно.

**Query-параметр API:** `open_now=true`

### 5.2 Высокий рейтинг

**Источник:** 2GIS API, поле `rating` (через `fields=items.reviews`).

**Логика:** двухэтапная:
1. Добавить `has_rating=true` к запросу 2GIS — убирает организации без рейтинга
2. Post-filter: оставить только `rating >= 4.5`

**Почему не только 2GIS-сортировка:** `sort=rating` сортирует, но не фильтрует. Салон с рейтингом 3.2 всё равно будет в выдаче — просто в конце. Фильтр «Высокий рейтинг» должен **убрать** всё ниже порога.

**Порог 4.5** — оптимален для бьюти: это хороший салон, но не «идеальные 5.0» (которых мало, и часть — накрутка). По мере роста платформы порог можно сделать динамическим (топ-30% по городу).

**Нюанс: чей рейтинг?**
На MVP — рейтинг 2GIS, потому что своих отзывов ещё нет. Когда появятся верифицированные отзывы платформы, нужно будет решить: показывать оба или приоритизировать свой. Рекомендация: показывать оба, но фильтровать по наивысшему из двух — это лояльнее к салонам.

**Query-параметр API:** `high_rating=true`

### 5.3 Онлайн-запись

**Источник:** наша БД, `salons.online_booking_enabled = true`.

**Логика:** показать только те результаты 2GIS, которые:
1. Имеют запись в `salon_external_ids` (т.е. зарегистрированы на платформе)
2. У связанного салона `online_booking_enabled = true`

**Это фильтр класса B** — резко сужает выдачу до зарегистрированных салонов. На раннем этапе с 5–10 салонами выдача будет почти пустой.

**UX-решение:** когда фильтр «Онлайн-запись» активен и результатов мало, показать блок:

> «Онлайн-запись пока доступна у [N] салонов рядом. Знаете салон, который стоит добавить? [Предложить салон]»

Это одновременно честная обратная связь и инструмент генерации лидов для онбординга салонов.

**Query-параметр API:** `online_booking=true`

### 5.4 Свободно сегодня

**Источник:** наша БД — `working_hours` + `appointments` + `services`.

**Логика:** салон считается «свободным сегодня», если:
1. Салон зарегистрирован на платформе
2. Сегодняшний `day_of_week` есть в `working_hours`
3. Между `opens_at` и `closes_at` существует хотя бы один слот длиной `min(services.duration_minutes)`, который **не перекрыт** существующими `appointments` со статусом `pending` или `confirmed`

**Упрощение для MVP:** проверять только наличие рабочего дня + отсутствие записей на весь день. Полный расчёт свободных слотов с учётом длительности услуг — следующая итерация.

```sql
-- Есть ли у салона рабочий день сегодня и хотя бы один незанятый час?
SELECT s.id
FROM salons s
JOIN working_hours wh ON wh.salon_id = s.id AND wh.day_of_week = :today_dow
WHERE s.id = ANY(:salon_ids)
AND s.online_booking_enabled = true
AND NOT EXISTS (
    -- Грубая проверка: считаем «свободным», если записей меньше, 
    -- чем рабочих часов (запас есть)
    SELECT 1 FROM appointments a
    WHERE a.salon_id = s.id
    AND a.starts_at::date = CURRENT_DATE
    AND a.status IN ('pending', 'confirmed')
    GROUP BY a.salon_id
    HAVING COUNT(*) >= EXTRACT(EPOCH FROM (wh.closes_at - wh.opens_at)) / 3600
);
```

**Это фильтр класса B** — работает только для зарегистрированных салонов.

**Бизнес-ценность:** это ключевая фича-дифференциатор. Фраза «Свободно прямо сейчас» в конкурентном анализе выделена как уникальная. Даже грубая реализация (есть/нет рабочий день + не полностью забит) создаёт ценность, которой нет у Яндекса и 2GIS.

**Query-параметр API:** `available_today=true`

### 5.5 До 2000 ₽

**Источник:** наша БД, таблица `services`.

**Логика:** салон проходит фильтр, если у него есть хотя бы одна активная услуга с ценой `price_cents <= 200000` (2000 ₽ × 100 копеек).

```sql
SELECT DISTINCT s.salon_id
FROM services s
WHERE s.salon_id = ANY(:salon_ids)
AND s.is_active = true
AND s.price_cents IS NOT NULL
AND s.price_cents <= 200000;
```

**Это фильтр класса B** — работает только для салонов с заполненными услугами.

**Нюанс про NULL:** `price_cents` nullable (цена «по договорённости»). Такие услуги **не проходят** фильтр — это корректно, потому что пользователь явно ищет предсказуемую цену.

**Будущее:** порог 2000 ₽ — hardcode для MVP. Потом можно сделать слайдер или несколько порогов (до 1000, до 2000, до 5000). Пороги лучше привязать к медианным ценам в категории — для маникюра 2000 ₽ это средний сегмент, для спа — бюджет.

**Query-параметр API:** `max_price=2000`

## 6. Итоговая схема запроса

### API контракт: `GET /api/v1/search`

| Параметр | Тип | Default | Описание |
|----------|-----|---------|----------|
| `lat` | float | 55.7558 (Москва) | Широта пользователя |
| `lon` | float | 37.6173 (Москва) | Долгота пользователя |
| `category` | string | — | `hair`, `nails`, `spa`, `barber`, `brows`, `makeup` |
| `sort` | string | `popular` | `popular`, `nearby`, `rating` |
| `open_now` | bool | false | Открыто сейчас (2GIS) |
| `high_rating` | bool | false | Рейтинг ≥ 4.5 |
| `online_booking` | bool | false | Только с онлайн-записью (платформа) |
| `available_today` | bool | false | Есть свободные слоты сегодня (платформа) |
| `max_price` | int | — | Макс. цена в рублях (платформа) |
| `page` | int | 1 | Страница |
| `page_size` | int | 20 | Размер страницы |
| `region_id` | int | — | ID региона 2GIS |

### Пайплайн обработки в SearchService

```
1. Принять и валидировать параметры

2. Определить класс запроса:
   has_platform_filters = online_booking || available_today || max_price != nil
   
3. Сформировать запрос к 2GIS:
   - sort → 2gis_sort
   - open_now → work_time=now
   - high_rating → has_rating=true
   - category → rubric_id / q
   - page_size:
     - без platform filters: 20 (обычный)
     - с platform filters: 50 (over-fetch)

4. Получить результаты 2GIS

5. Enrichment join:
   - Собрать external_ids из результатов
   - SELECT FROM salon_external_ids WHERE source='2gis' AND external_id IN (...)
   - Подтянуть salon данные + services

6. Post-filter (класс A):
   - high_rating: отбросить rating < 4.5

7. Post-filter (класс B, если has_platform_filters):
   - online_booking: оставить только enriched с online_booking_enabled
   - available_today: проверить working_hours + appointments
   - max_price: проверить services.price_cents
   
8. Если результатов < target И 2GIS имеет ещё страницы:
   - Повторить 4-7 со следующей страницей 2GIS
   - Макс. 3 итерации

9. Рассчитать расстояние (Haversine)

10. Вернуть результат
```

## 8. Порядок реализации

### Шаг 1 — Сортировка (быстрый win)

Минимальные изменения: добавить `sort` в query params, замапить на 2GIS `sort`. Фронтенд уже имеет `sortBy` в Redux.

Что трогаем:
- `SearchController.Search` — парсинг `sort`
- `SearchService.Search` — передача sort в `PlacesSearchInput`
- `PlacesSearchInput` модель — новое поле `Sort`
- `CatalogAdapter.SearchNearby` — маппинг на 2GIS `sort`
- `searchSlice` + `SearchBar` — подключение к API

**Оценка: 1 день на backend + frontend.**

### Шаг 2 — Фильтры класса A (Открыто сейчас, Высокий рейтинг)

Тоже пробрасываются в 2GIS. `work_time=now` и `has_rating=true` + post-filter по порогу рейтинга.

Что трогаем:
- `PlacesSearchInput` — поля `WorkTimeNow` (уже есть!), `HasRating`
- `CatalogAdapter` — пробросить в URL params
- `SearchService` — post-filter по рейтингу ≥ 4.5
- Frontend: активировать чипы `openNow`, `highRating` в `FilterRow`

**Оценка: 0.5–1 день.**

### Шаг 3 — Фильтры класса B (Онлайн-запись, До 2000 ₽)

Требуют enrichment join и post-filter. Over-fetch логика.

Что трогаем:
- `SearchService` — over-fetch loop (max 3 итераций)
- `SalonRepository` — метод `FindByExternalIDs` (уже есть), `FindServicesBySalonIDs` (уже есть)
- Post-filter функции: `filterOnlineBooking`, `filterMaxPrice`
- Frontend: активировать чипы, показать «мало результатов» UX

**Оценка: 1.5–2 дня.**

### Шаг 4 — Свободно сегодня

Самый сложный фильтр — нужна реальная проверка расписания и записей.

Что трогаем:
- Новый метод в `SalonRepository` или `AppointmentRepository`: `FindAvailableSalonIDs(salonIDs []uuid, date time.Time)`
- SQL-запрос: join `working_hours` + anti-join `appointments`
- Интеграция в post-filter pipeline

**Оценка: 1–1.5 дня.**

**Зависимость:** нужны реальные данные в `working_hours` — без них фильтр бессмысленен. Это упирается в онбординг салонов и dashboard для управления расписанием.

## 9. Комбинации фильтров: матрица поведения

| Комбинация | Поведение |
|-----------|----------|
| Только класс A | Обычный запрос, 2GIS фильтрует, пагинация стабильна |
| Только класс B | Over-fetch режим, результатов может быть мало |
| Класс A + класс B | 2GIS фильтрует (class A), потом post-filter (class B). Двойное сужение — самый жёсткий вариант |
| Все фильтры одновременно | Скорее всего 0–3 результата на раннем этапе. Это нормально — UX должен это отразить |

**Важное UX-правило:** при активных фильтрах класса B всегда показывать счётчик «N салонов с онлайн-записью» / «N салонов с ценами на платформе». Это: (a) даёт пользователю context, (b) мотивирует салоны регистрироваться — «почему моего салона нет в этом списке?»

## 10. Метрики успеха

| Метрика | Как измерить | Цель |
|---------|-------------|------|
| Использование сортировки | % сессий, где пользователь менял sort | > 15% |
| Использование фильтров | % сессий с хотя бы одним фильтром | > 25% |
| Конверсия фильтр → бронь | Бронирования из сессий с фильтрами vs без | Фильтры должны давать выше |
| «Мало результатов» показы | Сколько раз пользователь видит пустую/малую выдачу | Отслеживать для приоритизации онбординга |

## 11. Риски и ограничения

**Риск 1: Фильтры класса B бесполезны без онбординга.**
Если в БД 5 салонов, фильтр «Онлайн-запись» покажет 0–1 результат. Это неизбежная проблема стартового маркетплейса.
**Митигация:** не делать фильтры класса B дефолтно включёнными. Показывать их как опцию и честно отражать малое количество результатов.

**Риск 2: Over-fetch увеличивает latency.**
3 последовательных запроса к 2GIS × 200–400ms = 600–1200ms.
**Митигация:** лимит 3 итерации, показать первые результаты сразу (streaming), остальные догрузить. Или: сначала показать quick results (без platform filter), потом обновить с filter applied.

**Риск 3: Порог рейтинга 4.5 слишком жёсткий для некоторых категорий.**
В СПА может быть мало мест с 4.5+. В парикмахерских — навалом.
**Митигация на будущее:** динамический порог = P70 рейтинга в категории×регионе. На MVP — хардкод 4.5 достаточен.

**Риск 4: Демо-ключ 2GIS ограничен 50 результатами.**
Over-fetch на 3 страницах по 50 = 150 запрошенных, но демо-ключ не даст page > 5 и page_size > 10.
**Митигация:** для фильтров класса B нужен платный ключ. Без него over-fetch невозможен.
