# Поиск салонов: архитектура и поведение (актуальная реализация)

Документ описывает, **как сейчас устроен поиск** в монорепозитории beauty-marketplace: от запроса пользователя до ответа 2GIS и отрисовки списка. В конце зафиксированы недавние доработки (в т.ч. infinite scroll и UX загрузки).

---

## 1. Принципы (зачем так сделано)

### 1.1. 2GIS-first

- **Источник листинга** для главного сценария поиска — каталог 2GIS (`catalog.api.2gis.com`), а не собственная БД.
- **Почему:** на MVP в платформе мало своих салонов; пользователь должен видеть реальную выдачу рядом с ним.
- **Своя БД** используется как **слой обогащения**: если у филиала есть запись в нашей системе и связка по внешнему ID 2GIS, в карточку подмешиваются `salonId`, онлайн-запись, услуги и цены.

### 1.2. Сужение по рубрикам, а не «всё подряд»

- Запрос к 2GIS дополняется **`rubric_id`** — фиксированный набор ID рубрик «красоты» под выбранную категорию приложения (`hair`, `nails`, …).
- **Текстовый запрос `q`** всё ещё формируется из `categoryQuery` (например, «парикмахерская»), чтобы движок 2GIS получил и семантику, и жёсткую фильтрацию по рубрикам.
- **Почему:** уменьшить шум (случайные типы точек), приблизить выдачу к ожиданиям пользователя бьюти-приложения.

### 1.3. Пост-обработка на бэкенде

- 2GIS может вернуть объекты, которые по названию/рубрикам похожи на ТЦ, БЦ, ТРЦ. Такие точки **отфильтровываются** после ответа провайдера, с логированием причины.
- **Дедупликация по `externalId`** в рамках одной страницы ответа — чтобы не отдавать дубликаты карточек API.

### 1.4. Геоконтекст

- Поиск идёт в **окружности**: центр `point` (lat/lon из клиента или дефолт), **радиус 2000 м** на сервисе поиска.
- **Расстояние до пользователя** в карточке (`distanceKm`) считается на бэкенде по формуле гаверсинуса, **только если** в запросе переданы оба параметра `lat` и `lon` (см. контроллер).

### 1.5. Два режима на странице поиска

1. **Унифицированный листинг** (`GET /api/v1/search`) — то, что грузится при открытии страницы и смене категории/города: карточки `SearchResultCard`, infinite scroll.
2. **Текстовый поиск по каталогу** (отдельный API places) — после ввода строки в `SearchBar` и нажатия поиска: режим `PlaceCard`, пока есть результаты этого запроса.

Переключение: если `placesResult` непустой, UI показывает **режим places**, иначе — **режим unified search**.

## 2. HTTP API: унифицированный поиск

**Маршрут:** `GET /api/v1/search`

**Основные query-параметры:**

| Параметр     | Назначение |
|-------------|------------|
| `lat`, `lon`| Координаты центра поиска и (если оба заданы) для расчёта дистанции. Должны быть либо оба, либо ни одного. |
| `category`  | Категория приложения: `hair`, `nails`, `spa`, … Пустое/`all` на сервисе нормализуется к сценарию с дефолтной категорией. |
| `region_id` | ID региона 2GIS (нужен для `rubric_id` в запросе к каталогу). |
| `page`      | Номер страницы (с 1). |
| `page_size` | Размер страницы (ограничен на контроллере сверху, по согласованию с лимитами 2GIS). |

**Ответ:** JSON с полями `items` (массив карточек) и `total` (как сообщает 2GIS для этого запроса).

**Заголовки от клиента (для наблюдаемости):**

- `X-Client-Request-ID` — уникальный id запроса с UI.
- `X-Client-Action` — семантика действия, например `search_page_auto_refresh_manual`, `search_page_load_more_gps` (суффикс зависит от источника локации).

## 3. Бэкенд: цепочка от контроллера до 2GIS

### 3.1. `SearchController` (`internal/controller/search_controller.go`)

- Парсит `lat`/`lon`, `category`, `region_id`, `page`, `page_size`.
- Собирает `model.SearchInput` и вызывает `SearchService.Search`.

### 3.2. `SearchService` (`internal/service/search.go`)

1. **Центр поиска:** если в запросе нет пары lat/lon — используются координаты по умолчанию (Москва).
2. **Категория:** нормализация: пустая строка или `all` **заменяются на `hair`** (и для `q`, и для поля `Category` в `PlacesSearchInput`). Поэтому для маршрута `GET /api/v1/search` набор рубрик из ключа `all` в `categoryCoreRubrics` **не применяется** — берётся набор для `hair`. Широкий ключ `all` в адаптере остаётся для других потенциальных вызовов `SearchNearby` с `Category: "all"`.
3. Вызов **`PlacesService.SearchNearby`** с `PlacesSearchInput` (query, category, region, point, radius, page, locale).
4. **Фильтр «мусора»:** удаление элементов с подозрительными рубриками (`торговый центр`, `бизнес-центр`, …) или названиями (`тц`, `бц`, `трц`, …). В лог пишется `component=search_filter`, `drop_reason`.
5. **Дедуп по `externalId`** внутри уже отфильтрованного списка; при снятии дубля — лог с `drop_reason=duplicate_external_id`.
6. Сводный лог: `items_before_filter`, `items_after_filter`, `dedup_removed`, `spam_removed`.
7. **Обогащение из БД:** `FindByExternalIDs` по источнику `2gis`, затем услуги по салонам.
8. Сборка `SearchResultItem`: маппинг **категории UI из рубрик 2GIS** через таблицу `rubricCategory` (первое совпавшее имя рубрики); при отсутствии совпадений — fallback категории.
9. **Важно:** поле **`total` в ответе клиенту** сейчас берётся из **исходного** ответа 2GIS (`out.Total`), без пересчёта после фильтрации. Это значит, что при агрессивном отсеве «мусора» число «найдено» теоретически может не совпадать с длиной списка на странице — осознанный компромисс до отдельной доработки.

### 3.3. Адаптер 2GIS `CatalogAdapter` (`internal/infrastructure/twogis/catalog_adapter.go`)

**Эндпоинт:** `GET https://catalog.api.2gis.com/3.0/items`

**Ключевые параметры запроса:**

| Параметр | Значение / смысл |
|----------|------------------|
| `type` | `branch` — только филиалы организаций. |
| `search_type` | `one_branch` — на стороне 2GIS одна карточка на организацию в рамках логики движка (меньше дублей сетей). |
| `search_is_query_text_complete` | `true` — отключение префиксного матча по тексту (меньше ложных срабатываний). |
| `search_nearby` | `true` — усиление веса близости к точке. |
| `point`, `radius` | Круг поиска. |
| `page`, `page_size` | Пагинация; `page_size` на адаптере ограничен **10** (совместимость с демо-лимитами 2GIS). |
| `rubric_id` | Список ID через запятую из реестра **`categoryCoreRubrics`** по полю `Category` (если не переданы явные `RubricIDs`). |
| `region_id` | Обязателен для связки с рубриками, когда задан `rubric_id`. |
| `fields` | Минимально нужный набор: точка, отзывы, рубрики, расписание, флаги, атрибуты, полный адрес. |

**Опциональные флаги в модели `PlacesSearchInput`** (сейчас сервис поиска их не выставляет из HTTP, но адаптер готов): `HasRating`, `HasReviews`, `HasPhotos`, `WorkTimeNow` → соответствующие query-параметры 2GIS.

**Реестр рубрик (пример смысла, не догма навсегда):**

- `all` — широкий набор beauty-рубрик (парикмахерские, салоны, ногти, барбер, SPA, эпиляция, солярий, брови/ресницы, визаж и т.д.).
- Узкие категории — подмножества или пересечения (например, только `56759` для SPA).

ID рубрик привязаны к данным 2GIS; при смене справочника регионов их нужно перепроверять (см. `docs/2gis-api-reference-ru.md`).

## 4. Фронтенд: страница поиска и данные

### 4.1. Источник координат и города

Приоритет:

1. **Выбранный в UI город** из Redux (`locationSlice`): `lat`, `lon`, `regionId`, поле `source` (`manual` / `geo` / `storage`).
2. Иначе **браузерная геолокация** (`useGeolocation`): реальные координаты или fallback на центр Москвы, плюс `source`: `gps` или `default`.
3. Если lat/lon не ушли на API, **бэкенд** подставит свой дефолтный центр.

В `clientAction` добавляется суффикс `_manual`, `_geo`, `_storage`, `_gps`, `_default` — чтобы в логах отличать сценарии.

### 4.2. Хук `useSearch` (`frontend/src/shared/api/searchApi.ts`)

- Держит `data`, `isFetching`, `error`.
- **`search(params, { append })`:**
  - При **не-append** (новый запрос / сброс): перед запросом **`setData(null)`** — чтобы не показывать старую выдачу при смене категории или гео.
  - При **append**: предыдущий список сохраняется; в состояние дописываются только элементы с **новыми** `externalId` (защита от дублей между страницами).

### 4.3. Infinite scroll (`SearchPage.tsx`)

- При готовности гео и смене `lat`/`lon`/`regionId`/`category`/`locationSource` выполняется **первая загрузка** с `page=1`, счётчик страницы сбрасывается.
- Внизу списка — **sentinel** и `IntersectionObserver` с `rootMargin: 200px`; при пересечении вызывается `loadMore` с `page+1` и `{ append: true }`.
- Условие остановки: `items.length >= total` или уже идёт запрос.

### 4.4. Исправление UX: список не «мигает» при догрузке

**Проблема:** раньше флаг загрузки был `searchFetching` без учёта того, что данные уже есть. При append включался **полноэкранный** спиннер, и все карточки пропадали до ответа.

**Решение:**

- Полноэкранный лоадер только если **`searchFetching && searchResult === null`** (первая загрузка или явный сброс после replace).
- При догрузке страниц список остаётся на экране; индикатор загрузки может показываться **внизу** у sentinel (как в текущей разметке).

### 4.5. Сортировка «только с онлайн-записью»

- Фильтр `onlineOnly` **не скрывает** карточки без бронирования: список **пересортировывается** так, чтобы сначала шли салоны с `onlineBooking` (логика на клиенте в `displaySearchItems`).
