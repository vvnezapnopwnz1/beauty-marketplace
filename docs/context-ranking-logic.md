# Контекст: логика ранжирования салонов — итоги диалога

> Дата: апрель 2026 | Область: бизнес-логика выдачи списка салонов

---

## Суть задачи

Определить правила, по которым платформа показывает список салонов пользователю — так чтобы выдача воспринималась как релевантная и умная, даже на старте без платформенных данных.

## Скоринговая формула на будущее когда разберемся с апи

```
score =
  proximity_score    * 0.40
+ booking_enabled    * 0.25
+ external_rating    * 0.20
+ profile_quality    * 0.15
```

### Как считать каждый компонент (на будущее когда разберемся с апи)

**proximity_score** — нелинейный, с резким обрывом:

- до 1 км → 1.0
- 1–3 км → 0.7
- 3–5 км → 0.4
- 5+ км → 0.1

**booking_enabled** — бинарный: 1 или 0. Высокий вес (0.25) осознанный — салон с онлайн-записью конвертирует сессию в запись, это цель платформы.

**external_rating** — нормализуется к [0, 1]:

```
(rating - 3.0) / 2.0   // 3.0 → 0.0, 5.0 → 1.0
```

Рейтинг ниже 3.0 = 0, без дополнительного штрафа.

**profile_quality** — сумма заполненности профиля:

```
has_photo     → +0.4
has_services  → +0.4
has_price     → +0.2
```

## Жёсткие фильтры (до скоринга)

Исключаются из выдачи полностью:

- нет названия или адреса
- нет фото И нет услуг одновременно (пустая карточка хуже отсутствия)
- расстояние > 10 км (если пользователь не расширил поиск явно)

---

## UX-правила выдачи

- Первые 2–3 карточки — «витрина»: если они выглядят хорошо, пользователь оценивает весь список как качественный
- Диверсификация ценового диапазона в топе: не показывать только дорогие или только дешёвые
- Буст за `booking_enabled` не должен полностью перебивать близость и рейтинг — хороший салон в 300м без записи может быть выше среднего в 4км с записью, это нормально
- Платные/буст-позиции не ставить на первые 2 места без пометки «реклама»

## Связь с 2GIS API

Параметры, которые 2GIS позволяет использовать на стороне запроса (до получения данных):

| Параметр                                                      | Использование                                                     |
| ------------------------------------------------------------- | ----------------------------------------------------------------- |
| `rubric_id`                                                   | Фильтр по категории бьюти: `305,652,5603,110998,56759`            |
| `type=branch`                                                 | Только бизнес-точки, не здания и улицы                            |
| `point` + `radius`                                            | Геозона вокруг пользователя                                       |
| `has_photos=true`                                             | Предфильтр — только с фото (опционально)                          |
| `has_rating=true`                                             | Предфильтр — только с рейтингом (опционально)                     |
| `work_time=now`                                               | Фильтр «открыто сейчас» — основа для фичи «Свободно прямо сейчас» |
| `fields=items.reviews,items.point,items.schedule,items.flags` | Запросить рейтинг, координаты, расписание и флаги в одном запросе |

**Важно:** `sort=rating` или `sort=distance` от 2GIS не использовать — сортировка делается на нашей стороне по собственной формуле.

## Ограничения 2GIS, которые влияют на логику

- URL фотографий API не отдаёт — `has_photo` в скоринге это булевый флаг из `fields=items.flags`, не ссылка
- Контакты (телефон, email) — платная опция API, на MVP не используем
- Демо-ключ: максимум 50 результатов — для реальных данных нужен платный ключ
- Данные из API нельзя персистировать в БД (лицензия) — только realtime запросы

## Что осталось за рамками этого диалога

- Фазы 2 и 3 скоринга (платформенные данные, персонализация) — намеренно отложены
- Конкретная реализация JOIN между 2GIS ID и нашей таблицей `salon_external_ids`
- Логика расширения радиуса поиска при малом количестве результатов
